# 第八章 框架横向对比与选型

## Overview

> [!NOTE]
>
> 读完 Pi、Nanobot、OpenClaw、Hermes Agent 之后，我们应该把它们放在同一张图里看。
>
> 我们的判断是：
>
> **Nanobot 是一个很好的起点。**
>
> 它足够轻，能让初学者看清 Agent 的基础骨架；它又不至于太薄，已经包含 Provider、Runtime、Tool、Channel、Memory、WebUI 等完整个人助手所需的关键部件。
>
> 从 Nanobot 出发有三条路：
>
> ```text
> 向后退：拆解必要组件，退化到老祖宗 Pi，提炼真正的 Agent 内核
> 向前走：丰富功能与产品边界，演化到 OpenClaw，进入真实设备和多渠道
> 向上长：增强记忆、技能和自进化能力，走向 Hermes，让 Agent 越用越适应用户
> ```
>
> 所以本章不是要评出“谁最强”，而是回答：
>
> **不同阶段、不同目标、不同工程约束下，应该读哪个、用哪个、借鉴哪个？**

## 1. 一张总图

我们可以先把四个项目放在一个坐标系里：

```text
              Hermes Agent
        学习闭环 / Memory / Skills
                  ↑
                  │
初学者  ─────→  Nanobot  ──────→  OpenClaw
               好的起点           产品完整化
                  │
                  ↓
                  PI
              内核最小化
```

这张图表达的是：

- **Pi** 是往后退，把 Agent 拆到最小可运行内核。
- **Nanobot** 是中间点，轻量但完整，适合第一次理解 Agent 系统。
- **OpenClaw** 是往前走，把个人助手做成真实可用的本地平台。
- **Hermes** 是往上长，把 Agent 的经验沉淀、自我改进和用户适应做成闭环。

如果你不知道先读哪个，我们建议：

```text
先读 Nanobot
  ↓
觉得太复杂，就退回 Pi
  ↓
想做真实个人助手，就读 OpenClaw
  ↓
想做越用越懂你的 Agent，就读 Hermes
```

## 2. 四个项目分别适合什么

### 2.1 Pi：适合看 Agent 的最小内核

Pi 适合这些场景：

- 想理解一个 coding agent 最少需要哪些组件
- 想看模型、工具、状态、事件流如何组合
- 想做一个终端里的轻量 Agent harness
- 想把 Agent Runtime 嵌入到其他系统里
- 想学习如何把模型 Provider、Agent loop、TUI 拆成可复用包

Pi 不适合一上来就做完整个人助手。它更像一块薄而硬的底板。

它回答的问题是：

```text
如果只保留最必要的东西，一个 Agent 还剩什么？
```

### 2.2 Nanobot：适合做学习起点和轻量二次开发

Nanobot 适合这些场景：

- 第一次系统学习 Agent Framework
- 想看 LLM、ReAct、Runtime、Tool、Channel、Memory 的完整关系
- 想改一个轻量个人助手项目
- 想增加 Provider、Tool、Channel 或 WebUI 功能
- 想把 Agent 从“会说”扩展到“会做”

Nanobot 的优势是：它已经足够完整，但还没有复杂到像平台工程。

它回答的问题是：

```text
一个轻量个人助手系统，最核心的模块应该怎么组织？
```

这也是为什么我们认为 Nanobot 是好的起点。

### 2.3 OpenClaw：适合做真实个人助手平台

OpenClaw 适合这些场景：

- 想做运行在自己设备上的个人 AI 助手
- 想接入 WhatsApp、Telegram、Slack、Discord、iMessage 等真实渠道
- 想做长期运行的 Gateway
- 想支持本地文件、浏览器、语音、Canvas、移动端节点
- 想研究多渠道、多设备、多插件、多权限边界的 Agent 产品
- 想理解真实安全问题：DM pairing、allowlist、sandbox、doctor、配置迁移

OpenClaw 的重点不是“最小”，而是“真实”。

它回答的问题是：

```text
如果 Agent 要进入我的设备、聊天软件和本地权限系统，需要哪些工程层？
```

### 2.4 Hermes Agent：适合做学习型、长期陪伴型 Agent

Hermes 适合这些场景：

- 想让 Agent 越用越懂用户
- 想研究 memory、session recall、skills 的分层关系
- 想让 Agent 从任务经验中自动总结方法
- 想让用户偏好沉淀到后续任务行为里
- 想做会自我改进的长期运行 Agent
- 想研究“聊天记录如何变成可复用能力”

Hermes 的重点不是“接入多少平台”，而是“经验如何变成能力”。

它回答的问题是：

```text
Agent 如何把一次次对话，沉淀成下一次更可靠的行为？
```

## 3. 维度对比

| 维度 | Pi | Nanobot | OpenClaw | Hermes Agent |
| --- | --- | --- | --- | --- |
| 最核心定位 | 最小 coding harness | 轻量个人助手框架 | 本地优先个人助手平台 | 自我改进的学习型 Agent |
| 最适合读者 | 想看内核的人 | Agent 入门者 | 想做真实产品的人 | 想做长期记忆和技能沉淀的人 |
| 工程规模 | 小而薄 | 轻量完整 | 大型平台 | 大型学习型系统 |
| Agent Runtime | 清晰、核心化 | 自有 Runner / Loop | 外层编排 + Embedded Pi Runner | `AIAgent` 主循环 |
| Provider | 统一模型接口 | 多 Provider 抽象 | Provider plugin / catalog / auth | 多 Provider + 模型切换 |
| Tool | 核心 tool calling | 工具与 MCP | 工具、沙箱、权限、插件 | 大量工具与 toolsets |
| Channel | 以终端为主 | CLI / API / Channel / WebSocket | 多真实聊天渠道与设备节点 | CLI / Gateway / 多平台 |
| Gateway | 基本不强调 | 较轻 | 核心控制平面 | 有 messaging gateway |
| Memory | 不是重点 | 具备长期记忆思路 | 有 memory/plugin 能力 | 核心能力之一 |
| Skills | 资源扩展系统 | 可作为扩展能力 | 技能与插件生态 | 自我改进闭环核心 |
| Plugin | 包级模块化 | 较轻 | Plugin SDK / ClawHub / extensions | plugins + skills + memory providers |
| UI | TUI / Web UI | WebUI | Control UI + Apps + Canvas | CLI / TUI / Web / Gateway |
| 安全边界 | 较少 | 适中 | 强调 DM、安全、沙箱 | 强调工具审批、环境后端 |
| 适合生产落地 | 作为内核 | 小型助手和教学 | 个人助手平台 | 学习型助手和研究系统 |

## 4. 从 Nanobot 向后：回到 Pi

如果你读 Nanobot 时觉得模块很多，可以反向拆回 Pi。

Nanobot 里已经有：

- Provider
- Agent Runner
- Agent Loop
- Context Builder
- Tool Registry
- Channel
- WebUI
- Memory
- Bridge

但初学者很容易在这些产品能力里迷路，不知道什么是 Agent 的必要组件，什么只是产品增强。

这时候 Pi 的价值就出来了。

Pi 把问题压缩成：

```text
模型怎么接
消息怎么进
工具怎么调
状态怎么存
事件怎么发
终端怎么显示
```

从 Nanobot 退回 Pi，你会发现真正的内核大概是：

```text
Provider API
  ↓
Agent State
  ↓
Agent Loop
  ↓
Tool Calling
  ↓
Event Stream
  ↓
User Interface
```

所以 Pi 是“老祖宗”意义上的内核原型。

不是因为它功能最多，而是因为它把必要组件暴露得最干净。

## 5. 从 Nanobot 向前：走向 OpenClaw

如果你从 Nanobot 出发，想把它变成真实个人助手，就会自然走向 OpenClaw 的问题域。

Nanobot 已经有 channels，但真实世界的 channel 会变复杂：

- 谁可以私聊 Agent？
- 群聊里什么时候响应？
- 同一个用户在不同平台如何映射？
- 一个 channel target 如何对应 session？
- 工具执行结果如何发回原平台？
- 图片、语音、文件、thread、reaction 怎么处理？
- Agent 访问本地文件和浏览器时如何控制风险？

这些问题会把系统推向 OpenClaw 的形态：

```text
轻量入口
  ↓
多渠道接入
  ↓
Gateway 控制平面
  ↓
Session / Target / Account / Thread 路由
  ↓
Plugin SDK
  ↓
Companion Apps / Nodes / Canvas
  ↓
安全默认值与本地权限边界
```

所以 OpenClaw 是 Nanobot 的“产品化扩展方向”。

不是说 Nanobot 代码一定要变成 OpenClaw，而是说当你不断给 Nanobot 加真实能力时，迟早会遇到 OpenClaw 已经在处理的问题。

## 6. 从 Nanobot 向上：增强到 Hermes

Nanobot 已经能作为一个轻量个人助手运行，但如果你希望它越用越懂用户，就会遇到另一个问题：

```text
Agent 完成任务之后，经验去哪里？
```

如果只是保留聊天记录，下一次仍然可能忘。

如果只是向量检索，召回出来的还是“过去发生过什么”，不一定会改变未来做事方法。

Hermes 的关键是把经验分层：

```text
Memory：用户是谁，偏好什么
Session Recall：过去做过什么
Skills：以后这类任务应该怎么做
```

它的自进化闭环是：

```text
一次任务完成
  ↓
后台 review fork
  ↓
判断是否值得保存
  ↓
写入 memory 或 patch skill
  ↓
记录 provenance / usage
  ↓
下次同类任务直接继承经验
```

这就是从 Nanobot 向 Hermes 增强的方向：

- 给 Agent 加上后台复盘
- 区分用户偏好和任务技能
- 把一次经验沉淀成 class-level skill
- 记录技能是否被使用、查看、修改
- 避免把临时环境失败写成长期规则

Hermes 不是把 Agent 做得更“大”，而是让 Agent 有机会变得更“熟”。

## 7. 选型建议

### 7.1 如果你是初学者

先读 Nanobot。

理由是：它能让你一次看到 Agent 系统的主要部件，但不会马上被 Gateway、Plugin SDK、移动端、设备节点和复杂安全策略淹没。

学习路线：

```text
Nanobot 分层
  ↓
Agent Runner / Loop
  ↓
Context / Tool / Provider
  ↓
Channel / WebUI / Memory
```

### 7.2 如果你想理解最小内核

读 Pi。

尤其适合你想回答：

- Agent loop 到底是什么？
- Tool calling 如何进入状态机？
- TUI 和 Runtime 如何分离？
- 一个 Agent package 应该怎么拆？

Pi 适合当“骨架标本”读。

### 7.3 如果你想做个人助手产品

读 OpenClaw。

尤其适合：

- 本地优先部署
- 多聊天平台接入
- 移动端和桌面端联动
- Gateway / RPC / WebSocket 控制平面
- 安全边界和真实 DM 风险
- 插件生态和 SDK

OpenClaw 适合当“产品平台样板”读。

### 7.4 如果你想做长期陪伴和自进化

读 Hermes。

尤其适合：

- 用户画像
- 长期记忆
- 旧会话召回
- 技能自动更新
- 会话经验沉淀
- Agent 自我改进

Hermes 适合当“学习闭环样板”读。

### 7.5 如果你要做安全研究

建议重点读 OpenClaw 和 Hermes。

OpenClaw 的安全问题来自真实世界入口：

- 多 channel inbound
- DM pairing
- allowlist
- 本地工具权限
- 浏览器、文件、shell、账号凭据

Hermes 的安全问题来自长期学习：

- 错误经验是否会被永久保存
- 用户偏好是否会被过度泛化
- skill 是否可能被污染
- session recall 是否泄露旧会话
- background review 是否会把 transient failure 写成长期规则

这两类风险都很真实。

前者是“Agent 能接触真实世界”的风险。

后者是“Agent 会长期记住和改变自己”的风险。

### 7.6 如果你要做落地应用开发

建议从 Nanobot 起步，再按方向加能力。

```text
只做一个轻量助手：Nanobot
需要终端 coding 内核：参考 Pi
需要多平台个人助手：参考 OpenClaw
需要长期用户适应：参考 Hermes
```

这比一开始就选择最大项目更稳。

## 8. 本章小结

这四个项目可以组成一条非常清楚的学习路线：

```text
Pi
  看 Agent 最小内核

Nanobot
  看轻量完整骨架

OpenClaw
  看真实个人助手平台

Hermes
  看学习闭环和自我改进
```

我们的建议是把 Nanobot 放在中心。

因为它既不像 Pi 那样只保留内核，也不像 OpenClaw 和 Hermes 那样马上进入大型系统复杂度。

从 Nanobot 出发：

- 向后退，可以用 Pi 提炼真正的 Agent 内核。
- 向前走，可以用 OpenClaw 学习真实产品化能力。
- 向上长，可以用 Hermes 学习经验沉淀、自我改进和用户适应。

所以真正重要的不是“选一个项目信仰到底”，而是知道每个项目在 Agent 工程坐标系里的位置。

当你知道自己正在解决哪类问题，选型就会变得很自然。
