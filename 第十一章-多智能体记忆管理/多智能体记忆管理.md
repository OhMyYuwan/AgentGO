# 多智能体记忆管理

## Overview

> [!NOTE]
> **Workflow 决定 Agent 如何协作，Memory 决定 Agent 如何保存状态、复用经验并恢复任务。**
>
> 在单体 Agent 中，记忆管理通常只是“当前对话历史 + 工具调用结果”。  
> 但在多智能体系统中，问题会变得更复杂：多个 Agent 需要共享任务状态、传递中间结果、隔离局部上下文、避免记忆污染，并在长任务中支持回放、恢复和验证。
>
> 更直接一点说：
>
> **单体 Agent 的 Memory 解决“我刚才做了什么”；多智能体 Memory 解决“我们共同做到了哪一步、谁知道什么、下一步谁该接着做”。**
>
> 所以，多智能体记忆管理不是一个附属功能，而是 Agent 系统从“能跑一次”走向“能长期运行、可调试、可复现、可扩展”的基础设施。
>
> 本章按照常见 Workflow 分类展开，说明每类 Workflow 的协作结构、Memory 策略、适用场景，并为每类 Workflow 附上贴近日常工程、知识管理、内容生产、产品协作和自动化办公的例子。

## 1. 为什么多智能体需要记忆管理

在简单任务中，一个 Agent 可以把所有信息都放进上下文窗口里：用户目标、历史对话、工具输出、临时推理结果都在同一个上下文中完成。

但一旦进入多 Agent 场景，单纯依赖上下文窗口会遇到明显问题。

- **状态分散**：不同 Agent 掌握的信息不同，容易出现“每个 Agent 只知道局部”的情况。
- **上下文膨胀**：多个 Agent 的输出不断累积，完整塞进上下文会迅速超过窗口限制。
- **职责混乱**：写作 Agent 不一定需要知道全部数据清洗细节，排版 Agent 也不一定需要读取完整调研材料。
- **记忆污染**：一个 Agent 的错误结论可能被其他 Agent 继承并放大。
- **任务不可恢复**：长任务中断后，如果没有状态快照，就无法知道执行到哪一步。
- **结果不可复现**：如果没有保存执行轨迹，就无法复盘 Agent 为什么得出某个结论。

因此，多智能体系统需要把 Memory 从“对话历史”升级为“状态管理系统”。

| 单体 Agent | 多智能体系统 |
| --- | --- |
| 主要管理当前上下文 | 需要管理多个 Agent 的局部状态和全局状态 |
| 记忆通常随对话存在 | 记忆需要跨步骤、跨 Agent、跨任务持久化 |
| Debug 主要看单轮输出 | Debug 需要看任务状态、执行轨迹、Agent 交互 |
| 失败后通常重新开始 | 失败后应支持 checkpoint 恢复 |
| 上下文就是主要 Memory | Memory 需要分层、压缩、检索和权限隔离 |

## 2. Workflow 与 Memory 的关系

Workflow 和 Memory 是多智能体系统中的两条主线。

> **Workflow 解决“多个 Agent 如何协作”；Memory 解决“协作过程中产生的状态如何保存和复用”。**

如果只有 Workflow，没有 Memory，系统可以执行，但很难恢复、调试和复现。  
如果只有 Memory，没有 Workflow，系统保存了很多信息，但不知道谁在什么时候应该使用这些信息。

常见 Memory 类型包括：

| Memory 类型 | 作用 | 示例 |
| --- | --- | --- |
| Working Memory | 当前任务短期上下文 | 当前用户目标、当前工具输出 |
| Episodic Memory | 历史任务经验 | 之前失败过的写作策略、历史项目经验 |
| Semantic Memory | 长期知识 | 产品文档、业务规则、领域术语表 |
| Shared Memory | 多 Agent 共享状态 | 项目状态、任务进度、共享假设 |
| Tool Memory | 工具运行状态 | 数据处理任务、浏览器状态、外部 API 调用结果 |
| Reflection Memory | 失败反馈与反思 | 哪些方案无效、下一轮应如何调整 |
| Trace Memory | 完整执行轨迹 | Plan → Action → Observation → Retry |

## 3. Sequential Workflow：顺序流水线

Sequential Workflow 是最容易理解的多 Agent 协作模式。多个 Agent 按固定顺序依次执行，前一个 Agent 的输出成为后一个 Agent 的输入。

```text
Agent A → Agent B → Agent C → Result
```

### 3.1 典型结构

```text
资料收集 Agent
  ↓
内容整理 Agent
  ↓
初稿生成 Agent
  ↓
校对润色 Agent
  ↓
排版输出 Agent
```

这种模式适合步骤明确、依赖关系清晰、任务边界固定的场景。

### 3.2 Memory 策略

**Context Passing**  
前一个 Agent 的输出直接作为下一个 Agent 的输入。

```text
Agent A Output → Agent B Input → Agent C Input
```

优点是简单，缺点是错误会沿流水线传播。

**Shared Scratchpad**  
多个 Agent 共享一个草稿区，把阶段性结果写入同一个结构化状态中。

```json
{
  "task": "weekly report generation",
  "current_stage": "drafting",
  "materials": [],
  "outline": [],
  "open_questions": []
}
```

优点是状态统一，缺点是草稿区会不断膨胀，需要摘要和清理。

### 3.3 例子：周报生成流水线

```text
数据收集 Agent
  ↓
事项归类 Agent
  ↓
重点提炼 Agent
  ↓
周报撰写 Agent
  ↓
格式检查 Agent
```

| 阶段 | Agent | Memory 内容 |
| --- | --- | --- |
| 数据收集 | Data Collector | 日程、任务记录、聊天摘要、会议纪要 |
| 事项归类 | Organizer | 已完成事项、进行中事项、阻塞事项 |
| 重点提炼 | Summarizer | 本周亮点、关键进展、待协调问题 |
| 周报撰写 | Writer | 段落草稿、表达风格、受众要求 |
| 格式检查 | Formatter | 模板格式、标题层级、最终输出 |

这个例子中，Sequential Workflow 的优势是流程清晰，每一步都可以单独检查。  
缺点是如果数据收集阶段漏掉重要材料，后续 Agent 很难凭空恢复。

### 3.4 适用场景

- 文档生成流水线
- 周报、月报、会议纪要生成
- 数据清洗与摘要流程
- 简单自动化办公流程
- 固定模板内容生产

## 4. Hierarchical Workflow：层级调度

Hierarchical Workflow 采用 Manager-Worker 结构。Manager Agent 负责拆解任务、分配任务和汇总结果，Worker Agent 负责具体执行。

```text
Manager Agent
 ├── Research Agent
 ├── Writing Agent
 ├── Data Agent
 └── Review Agent
```

### 4.1 Memory 策略

**Centralized Memory**  
Manager 维护全局 Memory，Worker 只读取自己需要的上下文。

优点是全局一致性强，缺点是 Manager 容易成为瓶颈。

**Scoped Memory**  
每个 Worker 拥有自己的局部 Memory，Manager 只保存摘要和任务状态。

```text
Manager Summary Memory
 ├── Research Memory
 ├── Writing Memory
 ├── Data Memory
 └── Review Memory
```

优点是职责隔离清晰，缺点是跨 Agent 信息共享可能不足。

### 4.2 例子：产品需求分析团队 Agent

```text
Product Manager Agent
 ├── User Research Agent
 ├── Competitor Analysis Agent
 ├── Requirement Writing Agent
 ├── Data Analysis Agent
 └── Review Agent
```

| Agent | 职责 | Memory 内容 |
| --- | --- | --- |
| Product Manager | 拆分任务、控制方向、汇总结论 | 产品目标、用户群体、优先级、验收标准 |
| User Research Agent | 分析用户反馈和访谈材料 | 用户痛点、典型场景、原始反馈 |
| Competitor Analysis Agent | 对比同类产品 | 功能矩阵、差异点、优劣势 |
| Requirement Writing Agent | 生成需求说明 | 用户故事、功能范围、交互说明 |
| Data Analysis Agent | 分析指标和数据 | 使用数据、转化率、留存指标 |
| Review Agent | 检查一致性和遗漏 | 冲突点、缺失项、修改建议 |

这种结构适合复杂协作任务，因为不同 Agent 的职责边界比较清晰：有人负责调研，有人负责分析，有人负责写作，有人负责审查。

### 4.3 适用场景

- 产品需求分析
- 企业知识管理
- 研究报告生成
- 复杂内容生产
- 多角色协作任务

## 5. Planner-Executor Workflow：规划-执行分离

Planner-Executor Workflow 将“规划”和“执行”拆开。Planner 负责制定计划，Executor 负责执行动作，Verifier 负责验证结果。

```text
Planner Agent
  ↓
Executor Agent
  ↓
Verifier Agent
```

这种模式适合长任务，因为长任务往往需要先拆解，再逐步执行，并在执行过程中不断验证。

### 5.1 Memory 策略

**Task-Centric Memory**  
Memory 按任务组织，而不是按 Agent 组织。

```json
{
  "task_id": "market-research-001",
  "goal": "prepare a market research brief",
  "plan": [],
  "current_step": 3,
  "artifacts": [],
  "status": "running"
}
```

**Execution Trace Memory**  
保存完整执行轨迹。

```text
Plan → Action → Tool Call → Observation → Reflection → Retry
```

这种 Memory 非常适合 Debug 和复盘。

### 5.2 例子：市场调研自动化

```text
Planner Agent
  ↓ 制定调研计划
Search Agent
  ↓ 收集公开资料
Extraction Agent
  ↓ 提取公司、产品、价格和趋势
Analysis Agent
  ↓ 对比与归纳
Verifier Agent
  ↓ 检查引用和一致性
Writer Agent
  ↓ 输出调研简报
```

| Memory 类型 | 保存内容 | 作用 |
| --- | --- | --- |
| Task Memory | 调研目标、范围、交付格式 | 固定任务边界 |
| Source Memory | 检索来源、链接、摘录 | 支持引用与回溯 |
| Analysis Memory | 归纳结论、对比维度、指标 | 支持后续写作 |
| Trace Memory | 查询词、工具调用、观察结果 | 支持复盘 |
| Result Memory | 最终简报、图表、待确认问题 | 保存输出资产 |

### 5.3 适用场景

- 市场调研
- 资料收集与分析
- 长文档生成
- 自动化办公
- 多步骤数据处理
- 需要 checkpoint 的任务

## 6. Blackboard Workflow：黑板架构

Blackboard Workflow 中，多个 Agent 不直接通信，而是通过一个共享黑板读写状态。  
每个 Agent 都可以观察黑板上的事实、假设、证据和待处理任务，并根据自己的能力更新黑板。

```text
        Blackboard
       ↑    ↑    ↑
 Researcher Analyst Reviewer
```

### 6.1 Memory 策略

**Shared Global Memory**  
所有 Agent 读写同一个共享 Memory。

```json
{
  "facts": [],
  "hypotheses": [],
  "evidence": [],
  "pending_tasks": [],
  "decisions": []
}
```

这种策略适合异步协作，但必须处理状态冲突和版本控制问题。

常见控制方式：

- Versioned Memory：每次更新生成版本
- Event Sourcing：只追加事件，通过事件重建状态
- Lock：关键状态更新加锁
- Review Gate：关键结论必须经过审查 Agent 审核

### 6.2 例子：多源信息研究看板

```text
Literature Agent
News Agent
Data Agent
Interview Agent
Synthesis Agent
```

| 黑板区域 | 内容 |
| --- | --- |
| Facts | 已确认事实、来源、时间 |
| Hypotheses | 初步判断、待验证观点 |
| Evidence | 文献摘录、数据结果、访谈记录 |
| Pending Tasks | 待补充资料、待核对数据、待验证结论 |
| Decisions | 采用哪些结论、排除哪些材料 |

在这个场景中，每个 Agent 都把自己的发现写到黑板上。Synthesis Agent 不是从零开始分析，而是基于黑板中的结构化材料进行综合。

### 6.3 适用场景

- 多源资料研究
- 复杂选题分析
- 团队知识看板
- 异步协作写作
- 跨部门信息汇总

## 7. Debate Workflow：辩论式协作

Debate Workflow 让多个 Agent 从不同角度分析同一个问题，再由 Judge Agent 汇总结论。

```text
Agent A：支持观点
Agent B：反对观点
Agent C：补充证据
        ↓
Judge Agent
```

这种模式适合不确定性高、需要多视角判断的任务。

### 7.1 Memory 策略

**Independent Memory**  
每个 Agent 独立推理，避免被其他 Agent 影响。

**Argument Memory**  
只共享论点、证据、反驳和裁决结果。

```json
{
  "claim": "this product direction is feasible",
  "evidence": [],
  "counter_arguments": [],
  "judge_decision": ""
}
```

Argument Memory 的价值在于保留“可审查证据”，而不是保存所有冗长推理。

### 7.2 例子：产品方向评审

```text
Opportunity Agent：论证机会存在
Constraint Agent：提出资源和时间限制
User Value Agent：评估用户价值
Business Agent：评估商业价值
Judge Agent：综合裁决
```

| Agent | 立场 | Memory 内容 |
| --- | --- | --- |
| Opportunity Agent | 寻找支持理由 | 市场需求、用户反馈、趋势材料 |
| Constraint Agent | 提出限制条件 | 成本、周期、依赖、组织资源 |
| User Value Agent | 评估用户收益 | 用户痛点、使用频率、替代方案 |
| Business Agent | 评估业务收益 | 收入影响、增长机会、优先级 |
| Judge Agent | 汇总判断 | 证据表、决策建议、置信度 |

这种模式可以减少单个 Agent 过度乐观或过度保守的问题。

### 7.3 适用场景

- 产品方向评审
- 战略方案比较
- 投资可行性分析
- 复杂决策支持
- 高不确定性判断

## 8. Swarm Workflow：群体协同

Swarm Workflow 由大量 Agent 以去中心化方式协同完成任务。  
每个 Agent 通常只掌握局部信息，通过消息传播、信号扩散或局部规则形成整体行为。

```text
Agent Agent Agent Agent Agent
   ↘   ↓   ↙   ↘   ↓
      Emergent Coordination
```

### 8.1 Memory 策略

**Distributed Memory**  
每个 Agent 只保存局部状态。

**Emergent Shared Memory**  
没有显式中心 Memory，而是通过交互形成群体状态，例如投票结果、声誉分数、局部发现传播和多 Agent 共识。

### 8.2 例子：大规模资料筛选

```text
Keyword Agent 群
Source Agent 群
Summary Agent 群
Deduplication Agent 群
Ranking Agent 群
```

| Agent 群 | 任务 | Memory 内容 |
| --- | --- | --- |
| Keyword Agent | 扩展检索关键词 | 关键词、同义词、主题标签 |
| Source Agent | 收集候选资料 | 链接、标题、来源质量 |
| Summary Agent | 摘要候选资料 | 摘要、核心观点、引用片段 |
| Deduplication Agent | 去重和合并 | 相似材料、重复来源、合并结果 |
| Ranking Agent | 评估材料价值 | 相关性、新颖性、可信度 |

Swarm Workflow 的优势是扩展性强，适合大量目标并发处理。  
缺点是全局一致性弱，需要额外的结果聚合和去重机制。

### 8.3 适用场景

- 大规模资料筛选
- 众包式内容标注
- 多候选方案搜索
- 分布式任务处理
- 多智能体模拟实验

## 9. Graph Workflow：图状态调度

Graph Workflow 将任务建模为图。节点可以是 Agent、工具或函数，边表示状态转移条件。

```text
Node = Agent / Tool / Function
Edge = Condition / State Transition
```

例如：

```text
Start
  ↓
Classify
  ├── if question → Answer
  ├── if document task → Draft
  └── if uncertain → HumanReview
```

### 9.1 Memory 策略

**Graph State Memory**  
Memory 被挂载到图状态中。

```json
{
  "global_state": {},
  "node_states": {},
  "edge_conditions": {},
  "checkpoints": []
}
```

Graph State Memory 的关键价值是：支持分支、循环、回退和恢复。

### 9.2 例子：客服工单处理流程

```text
接收工单
  ↓
意图分类
  ├── 账号问题 → Account Agent
  ├── 账单问题 → Billing Agent
  ├── 使用问题 → Help Agent
  └── 不确定 → Human Review
  ↓
生成回复
  ↓
满意度确认
  ├── 满意 → 关闭
  └── 不满意 → 升级处理
```

| 节点 | 状态内容 |
| --- | --- |
| 接收工单 | 用户问题、渠道、时间 |
| 意图分类 | 问题类型、置信度、分流理由 |
| 专项处理 | 对应业务知识、处理步骤、结果 |
| 人工复核 | 复核原因、人工结论 |
| 生成回复 | 回复草稿、引用资料、语气要求 |
| 关闭工单 | 最终状态、满意度、后续动作 |

这种结构适合有明确状态流转、分支条件和回退需求的业务流程。

### 9.3 适用场景

- 客服工单系统
- 企业审批流程
- 内容发布流程
- 项目任务流转
- 长周期状态机任务

---

## 10. Event-Driven Workflow：事件驱动

Event-Driven Workflow 以事件为核心。Agent 不按固定顺序执行，而是订阅特定事件，在事件触发时执行动作。

```text
Event Bus
 ├── New Email Event → Email Agent
 ├── Meeting Event → Calendar Agent
 ├── File Updated Event → Document Agent
 └── User Request Event → Assistant Agent
```

### 10.1 Memory 策略

**Event Memory**  
系统记录事件流，再根据事件回放或重建状态。

```json
{
  "event_id": "evt-001",
  "type": "meeting_finished",
  "timestamp": "2026-01-01T10:00:00Z",
  "payload": {},
  "handled_by": []
}
```

**Subscription Memory**  
记录每个 Agent 订阅什么事件、处理过哪些事件。

```json
{
  "agent": "calendar-agent",
  "subscriptions": ["meeting_created", "meeting_finished"],
  "processed_events": []
}
```

### 10.2 例子：会议后自动整理

```text
Meeting Finished Event
  ↓
Transcript Agent
  ↓
Summary Agent
  ↓
Action Item Agent
  ↓
Calendar Agent
  ↓
Email Draft Agent
```

Memory 设计：

- Event Memory 保存会议结束、转录完成、待办生成等事件
- Transcript Memory 保存会议转写和发言人信息
- Action Memory 保存待办事项、负责人和截止日期
- Result Memory 保存摘要、邮件草稿和后续提醒

### 10.3 适用场景

- 自动会议纪要
- 邮件与日程自动化
- 文件变更通知
- 实时监控与提醒
- 长期运行 Agent 系统

---

## 11. Reflection Workflow：反思迭代

Reflection Workflow 通过“生成—批评—修正—重试”的循环提高结果质量。

```text
Generate
  ↓
Critique
  ↓
Refine
  ↓
Retry
```

这种模式适合需要反复试错的任务。

### 11.1 Memory 策略

**Reflection Memory**  
保存失败原因、反思结论和下一步策略。

```json
{
  "failure": "the first draft is too generic",
  "reflection": "the draft lacks concrete examples and audience-specific wording",
  "next_strategy": "add examples and rewrite for managers"
}
```

Reflection Memory 的关键不是“保存所有失败”，而是保存能指导下一轮行动的失败经验。

### 11.2 例子：演讲稿迭代优化

```text
Draft Agent
  ↓ 生成初稿
Critique Agent
  ↓ 评价结构、语气和内容密度
Refine Agent
  ↓ 根据反馈改写
Audience Agent
  ↓ 从听众视角检查是否清晰
```

| Memory 类型 | 内容 |
| --- | --- |
| Draft Memory | 每轮草稿版本 |
| Critique Memory | 问题点、修改建议 |
| Audience Memory | 受众画像、理解难点 |
| Reflection Memory | 哪类表达无效、下一轮如何调整 |
| Strategy Memory | 改写策略、风格要求 |

例如，某一轮失败后，Reflection Agent 可能总结：

> 当前版本概念解释过多，但缺少与听众工作场景的连接。下一轮应减少抽象概念，增加具体业务例子。

### 11.3 适用场景

- 演讲稿优化
- 文档润色迭代
- 教学材料改写
- 产品方案打磨
- Agent 自我改进

---

## 12. Tree-of-Thought Workflow：思维树搜索

Tree-of-Thought Workflow 将推理过程展开为多条分支，每条分支代表一种可能的思路、计划或假设。

```text
Root Thought
 ├── Thought A
 │   ├── A1
 │   └── A2
 └── Thought B
     ├── B1
     └── B2
```

### 12.1 Memory 策略

**Branch-Isolated Memory**  
每条推理分支维护独立 Memory，避免不同思路互相污染。

```text
Branch A Memory
Branch B Memory
Branch C Memory
```

常见优化方法：

- Pruning：剪掉低价值分支
- Shared Prefix Memory：共享公共前缀状态
- Heuristic Retrieval：只保留关键节点
- Beam Search：只保留评分最高的若干分支

### 12.2 例子：活动策划方案搜索

```text
Root：设计一场技术分享活动
 ├── Branch A：小型闭门研讨会
 ├── Branch B：公开线上直播
 ├── Branch C：线下 Workshop
 └── Branch D：系列课程形式
```

| 分支 | 假设 | Memory 内容 |
| --- | --- | --- |
| Branch A | 小范围高质量交流 | 目标人群、议题深度、邀请名单 |
| Branch B | 扩大传播范围 | 直播平台、宣传渠道、互动方式 |
| Branch C | 强调实操体验 | 教室、设备、练习材料 |
| Branch D | 长期内容沉淀 | 课程大纲、节奏安排、复用材料 |

Tree-of-Thought 的优势是适合探索复杂方案。  
缺点是分支容易爆炸，因此必须配合剪枝、评分和证据验证。

### 12.3 适用场景

- 复杂方案设计
- 多路径规划
- 活动策划
- 策略比较
- 数学与逻辑推理

---

## 13. 通用应用中的组合架构

单一 Workflow 往往只能解决一类问题。  
在真实应用中，内容生产、项目协作、知识问答和自动化办公通常需要组合多个 Workflow。

### 13.1 推荐组合一：Planner-Executor + Reflection

适合：

- 长文档生成
- 市场调研
- 演讲稿优化
- 项目计划制定

结构：

```text
Planner Agent
  ↓
Executor Agent
  ↓
Verifier Agent
  ↓
Reflection Agent
  ↺
```

这个组合适合“先计划、再执行、再检查、失败后反思”的任务。

### 13.2 推荐组合二：Hierarchical + Debate

适合：

- 产品方案评审
- 战略分析
- 多部门协作
- 高不确定性决策

结构：

```text
Manager Agent
 ├── Opportunity Agent
 ├── Constraint Agent
 ├── Data Agent
 └── Judge Agent
```

这个组合可以同时保留组织分工和多角度审查能力。

### 13.3 推荐组合三：Graph + Event-Driven

适合：

- 客服工单
- 审批流程
- 会议后自动跟进
- 长周期项目管理

结构：

```text
New Event
  ↓
Graph Workflow Instance
  ↓
Classify → Process → Review → Notify → Archive
```

事件负责触发流程，图状态负责管理流程推进。

### 13.4 通用 Memory 分层设计

| Memory 层 | 保存内容 | 作用 |
| --- | --- | --- |
| Global Task Memory | 目标、范围、交付格式、验收标准 | 固定任务边界 |
| Source Memory | 原始材料、链接、摘录、数据表 | 支持引用和回溯 |
| Working Memory | 当前步骤的输入、输出和临时判断 | 支持即时推理 |
| Execution Trace Memory | 工具调用、命令、观察结果 | 支持复盘和恢复 |
| Reflection Memory | 失败原因、下一步策略、经验总结 | 指导下一轮改进 |
| Retrieval Memory | 历史项目、知识库、模板和案例 | 提供长期知识 |

---

## 14. Workflow 与 Memory Strategy 速查表

| Workflow | 常见 Memory Strategy | 典型例子 | 优点 | 缺点 |
| --- | --- | --- | --- | --- |
| Sequential | Context Passing / Shared Scratchpad | 周报生成流水线 | 简单、可控 | 错误级联、上下文膨胀 |
| Hierarchical | Centralized / Scoped Memory | 产品需求分析团队 Agent | 分工清晰、易管理 | Manager 瓶颈、摘要丢失 |
| Planner-Executor | Task-Centric / Execution Trace | 市场调研自动化 | 易恢复、可回放 | Trace 成本高 |
| Blackboard | Shared Global Memory | 多源信息研究看板 | 异步解耦 | 冲突和污染严重 |
| Debate | Independent / Argument Memory | 产品方向评审 | 多视角、降低片面性 | 成本高、重复推理 |
| Swarm | Distributed / Emergent Memory | 大规模资料筛选 | 扩展性强 | 一致性差 |
| Graph | Graph State Memory | 客服工单处理流程 | 支持分支和恢复 | 设计复杂 |
| Event-Driven | Event / Subscription Memory | 会议后自动整理 | 可追溯、适合异步 | 事件量大 |
| Reflection | Reflection Memory | 演讲稿迭代优化 | 持续改进 | 错误反思可能被强化 |
| Tree-of-Thought | Branch-Isolated Memory | 活动策划方案搜索 | 多路径探索 | 分支爆炸 |

# 本章小结

多智能体系统的核心不是简单增加 Agent 数量，而是设计合适的 Workflow，并为每类 Workflow 配套合适的 Memory Strategy。

可以用一句话概括：

> **Workflow 决定 Agent 如何协作，Memory Strategy 决定 Agent 如何保存状态、复用经验、避免漂移并支持长期运行。**

不同任务适合不同的 Workflow：

| 任务特征 | 推荐 Workflow |
| --- | --- |
| 步骤固定、流程短 | Sequential |
| 角色分工明确 | Hierarchical |
| 任务可拆解、需要执行验证 | Planner-Executor |
| 多源信息异步汇聚 | Blackboard |
| 需要多角度判断 | Debate |
| 大规模并发任务 | Swarm |
| 状态复杂、分支明确 | Graph |
| 事件实时触发 | Event-Driven |
| 需要不断试错改进 | Reflection |
| 需要探索多条推理路径 | Tree-of-Thought |

对于通用 Agent 应用，推荐采用组合式架构：

```text
Planner-Executor 作为主流程
Reflection 负责失败反馈
Tree-of-Thought 负责多路径探索
Debate 负责多角度判断
Graph State 负责全流程管理
Event-Driven 负责外部事件触发
```

这样既能支持自动化执行，也能保留可解释性、可回放性和任务可恢复性。
