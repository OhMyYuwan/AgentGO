# 🧠 Agent Memory 体系全解析

> 理解 AI Agent 的记忆机制，是构建更智能、更个性化 Agent 的基础。
> 本文拆解为：**短时记忆 → 长时记忆 → 两者配合 → 检索记忆 → 用户建模**
> 
> ⚠️ **分类说明**：四分类是本文为分析便利而设立的**工程实践框架**，并非学界共识分类。在主流文献（如 LangChain、MemGPT、Generative Agents）中，"检索记忆（RAG）"通常不被视为一类独立的记忆类型，而是**访问长时记忆的一种机制**；长时记忆本身则常被细分为情节记忆（Episodic）、语义记忆（Semantic）和程序性记忆（Procedural）。读者在学术引用时请留意这一差异。

---

## 一、🟡 短时记忆（Short-Term Memory / Context Window）

### 1.1 本质
短时记忆是 Agent 在**当前推理过程中可直接访问的即时状态信息**。技术实现上，它由 Context Window（上下文窗口）承载，包含用户消息、助手回答、工具调用结果等完整对话记录。会话开始时，系统可预加载基础上下文（如 system prompt、用户画像），此后的每轮对话实时追加；会话结束后，窗口内的所有对话状态随之清空。一个直观的类比是：每开启一个新会话窗口，Agent 就像拿到了一块**全新的"工作记忆板"**——之前的笔记全部擦除，一切从零开始。

### 1.2 两层架构

Agent 短时记忆含两层：上层为 Agent 框架的**主动管理**（筛选、裁剪、压缩），下层为 LLM Context Window 的**被动承载**（接收 Prompt，溢出丢弃）。前者决定"装什么"，后者提供"装在哪"。

### 1.2.1 下层：LLM Context Window（被动承载）

```
用户输入 A → 用户输入 B → 用户输入 C
        ↓           ↓           ↓
  [A, B, C] 被拼接成完整 Prompt → 送入 LLM
```

- **默认全量携带**：所有历史消息拼接为完整 Prompt，不做筛选
- **被动溢出**：Token 满了就丢弃最早的，不主动判断保留谁、丢弃谁
- **顺序感知**：位置编码使 LLM 能感知消息先后顺序
- **非主动检索**：不查知识库，直接读 Prompt 里的内容

#### 例子

```
第一轮 ─ 用户问："这篇论文的创新点是什么？"
         当前 Prompt = [用户: "这篇论文的创新点是什么?"]
                       → LLM 生成回答："创新点是 A、B、C……"

第二轮 ─ 用户问："那实验设置呢？"
         当前 Prompt = [用户: "创新点是什么?"
                       助手: "创新点是 A、B、C……"
                       用户: "那实验设置呢?"]
                       → LLM 生成回答（直接基于上文理解实验设置）
```

第二轮时第一轮的 Q&A 还在窗口里，LLM 直接读，**全程没有检索**。

#### Token 限制

```
Context Window（如 128K tokens）
[最早 A] ← 被挤出  [B] [C] ... [最新]
      窗口满了，最早的内容滚动丢弃
```

| 保留时长 | 容量上限 | 溢出策略 |
|----------|---------|---------|
| 仅当前会话，关窗即失 | 受模型 Token 限制（GPT-4o: 128K, Claude: 200K） | 滑动窗口截断、摘要压缩、重要消息保留 |

### 1.2.2 上层：Agent 主动管理策略

| 策略 | 做什么 | 例子 |
|------|-------|------|
| **筛选/裁剪** | 移除与当前任务无关的对话，释放 Token | 砍掉闲聊天气，保留技术讨论 |
| **摘要压缩** | 多轮对话压缩为一段摘要，代替原始记录 | 8 轮讨论 → "用户正在用 FastAPI 搭建后台，已完成 JWT 登录" |
| **分层调度** | 核心上下文在窗口，次要上下文暂存外部，按需换入 | MemGPT：主窗口 ↔ 外部归档，类似 OS 虚拟内存 page-in/out |
| **重排/优先级** | 重要消息放在 Prompt 关键位置，System Prompt 固定在最前作为锚点 | 用户偏好、关键决策优先排列 |

### 💡 实践建议
> 一个窗口只问同一类问题。跨话题的上下文会互相"污染"，导致模型回答偏移，且白白消耗 token 额度。

## 二、🔵 长时记忆（Long-Term Memory / External Knowledge Base）

### 2.1 本质

长时记忆是**持久化存储**的知识，脱离单次会话独立存在。它解决了"关闭窗口就忘光"的问题。

> 本节从三个方面展开：**① 写入途径**（记忆怎么进来）→ **② 存储形式**（存在哪、以什么结构存）→ **③ 核心用途**（这些记忆最终用来解决什么问题）。

### 2.2 写入途径

长时记忆不是只有一种来源。按"知识怎么进入记忆系统"来分，至少有以下五种途径：

| 途径 | 说明 | 例子 |
|------|------|------|
| **对话蒸馏** | 对话过程中自动提取高价值信息写入存储 | 用户多次提到 FastAPI → Agent 记录"技术栈: FastAPI" |
| **反思生成** | Agent 定期对已有记忆做推理，提炼更高阶洞察 | 10 条零散记录 → "用户的性能瓶颈在数据层，非前端" |
| **外部知识注入** | 人工/脚本将文档批量导入知识库 | 把公司 Wiki 全部 Embedding 后写入 Vector DB |
| **工具/环境反馈** | 每次工具调用或代码执行的结果自动沉淀 | API 返回 401 → 记住"这个接口需要 Bearer Token" |
| **用户显式标记** | 用户直接说"记住这个"，Agent 赋予最高保留权重 | "记住我部署用的是 AWS ECS 不是 EC2" |

### 2.3 存储形式

不管通过哪种途径写入，最终都落在**LLM 之外的持久化存储**中。不同性质的记忆走不同后端、以不同结构存：

| 存储后端 | 存什么 | 存的形式 | 用途 |
|---------|-------|---------|------|
| **Vector DB**（Pinecone / Chroma） | 知识片段、对话摘要 | `chunk` + `embedding 向量` + metadata | "意思相近"的模糊检索 |
| **KV Store**（Redis） | 用户画像、偏好配置 | `user:u_001:tech_stack` → `{ "backend": "FastAPI" }` | 会话启动时秒级加载 |
| **图数据库**（Neo4j） | 实体关系 | `(u_001)-[:USES]->(FastAPI)-[:DEPLOYS_ON]->(AWS)` | 跨实体多跳推理 |
| **关系型数据库** | 对话日志、审计记录 | 行记录：`id / user_id / content / timestamp` | 精确查询、按时间回放 |

> 📌 四者互补，一个成熟的 Agent 记忆系统通常同时使用其中 2-3 种。

### 2.4 核心用途

回到最根本的问题：**这些持久化记忆，最终解决什么现实问题？**

| 用途 | 说明 | 例子 |
|------|------|------|
| **跨会话连续性** | 下次对话时，Agent 还记得上次聊到什么、做到哪一步 | 用户昨天讨论了 API 设计方案，今天打开窗口继续，Agent 自动回顾上下文 |
| **个性化适配** | 积累用户偏好、习惯、知识水平，让回答越来越"懂你" | Agent 记得用户喜欢表格而非段落，是 React 专家，自动调整回答格式和深度 |
| **知识积累** | 把零散的信息聚合成结构化知识，越用越"聪明" | 从 10 次对话中提取出用户的技术栈、项目架构、常用工具链 |
| **错误避免** | 记住过往的错误和用户纠正，不再重复 | 用户说过"不要用 fetch，我们项目用 axios"→ Agent 后续建议都用 axios |
| **任务恢复** | 长时间任务中断后，能从中断点继续，而非从头开始 | 一个持续 3 天的代码重构，每天 Agent 都能从上次的进度接着干 |

一句话：**短时记忆管"这一轮"回答得好不好，长时记忆管"下一次"Agent 还认不认识你。**

## 三、🟠 短时记忆与长时记忆的配合

长时记忆不是 LLM 在生成回答时"现场去翻"的——它发挥作用的时间点在**每轮对话开始前的预处理阶段**。用一个具体例子串起来：

```
用户问："帮我写一个 API 接口"

   │
   ▼
① Agent 从长时记忆中取出相关上下文（这一步发生在 LLM 生成之前）：
   ├── 用户偏好：喜欢 FastAPI、代码简洁、用 axios 不用 fetch
   ├── 项目事实：上周讨论过 User 和 Order 两个数据模型
   └── 历史纠正：上次用户说"别用 ORM，直接写 SQL"
   │
   ▼
② 把这些 + 用户当前问题，拼接成一份完整的 Prompt：
   "你是项目助手。
    [已知] 上周讨论了 User、Order 模型。
    [偏好] 用户喜欢 FastAPI + 直接写 SQL + axios。
    用户问：帮我写一个 API 接口"
   │
   ▼
③ 这个 Prompt 进入短时记忆（Context Window），LLM 读它，生成回答
   │
   ▼
④ 本轮对话结束后，如果出现新信息（比如用户说"端口用 8080"），
   Agent 蒸馏并写回长时记忆
```

> 📌 **一句话**：长时记忆在"拼 Prompt"之前干活，短时记忆在"拼好 Prompt"之后干活，LLM 只管读 Prompt 生成答案。三者各管一段，互不越界。

## 四、🟢 检索记忆（Retrieval Memory / RAG）

### 4.1 本质
检索记忆是**"用时才取"**的机制：不把所有知识塞进上下文，而是在需要时精准检索相关片段。

> 📝 **与长时记忆的关系**：在 LangChain 等主流框架中，检索记忆（RAG）和长时记忆并非并列的两类记忆——RAG 本质上是**访问长时记忆的检索机制**。长时记忆是"存了什么"，检索记忆是"怎么取"。本文将二者并列讨论，是为了工程技术上的可操作性，而非学术分类。

### 4.2 基础 RAG 流程

```
用户提问（Query）
      ↓
  Embedding 模型
      ↓
   Query 向量
      ↓
   向量数据库相似度检索（Top-K）
      ↓
  召回相关文档片段（Chunks）
      ↓
  拼入 Prompt → LLM 生成回答
```

### 4.3 检索增强的主要范式

#### 🔹 Naive RAG（基础 RAG）
- 直接对文档切块 → embedding → 检索 → 生成
- 问题：切块破坏语义，跨段落推理能力弱

#### 🔹 Advanced RAG
在基础 RAG 上增加：
- **查询改写**（Query Rewriting）：用 LLM 把用户问题改成更适合检索的形式
- **混合检索**：向量检索 + 关键词检索（BM25）结合
- **重排序**（Reranking）：用 Cross-Encoder 对召回结果重新打分排序

#### 🔹 GraphRAG（微软，2024）
```
文档 ──→ 实体抽取 ──→ 构建知识图谱
                           ↓
                    图遍历 + 社区摘要
                           ↓
                    结构化上下文 → LLM
```
- 优势：能处理需要**跨文档、多跳推理**的问题（如"A 和 C 的关系是什么"）
- 代价：构建成本高，适合知识密集型场景

#### 🔹 HyDE（假设文档嵌入）
- 先让 LLM 生成一个"假设答案"，再用这个假设答案去检索，提升召回质量

### 4.4 检索质量的关键指标

| 指标 | 含义 |
|------|------|
| 召回率（Recall） | 相关文档有没有被找到 |
| 精确率（Precision） | 找到的文档有多少是真正相关的 |
| 答案忠实度（Faithfulness） | 生成的答案有没有"幻觉"，是否基于检索到的内容 |

## 五、🟣 用户建模（User Modeling / Personalization Memory）

### 5.1 本质
用户建模是 Agent 对**特定用户**的持续理解与画像构建。它不是在回答"世界是什么"，而是在回答**"这个用户是谁，他想要什么"**。

### 5.2 建模的核心维度

```
┌─────────────────────────────────────────────────────┐
│                   用户画像 Profile                   │
│                                                     │
│  🎯 目标与意图    → 用户当前想做什么、长期目标是什么  │
│  💡 知识水平      → 专家/初学者？偏好技术深度还是概述 │
│  ❤️ 偏好与风格    → 喜欢简洁还是详细？表格还是文字？  │
│  🔄 行为模式      → 常问哪类问题？习惯什么交互节奏？  │
│  📌 长期事实      → 职业、使用语言、所在时区等        │
└─────────────────────────────────────────────────────┘
```

### 5.3 建模的技术路径

#### 路径 1：显式收集
用户主动告知（如注册时填写偏好、对话中直接说"我是前端开发者"）

#### 路径 2：隐式推断
从用户行为中自动提取：

```
用户连续问了 5 个 React 问题
         ↓
Agent 推断：用户是前端开发者，偏好 JavaScript 生态
         ↓
存入用户画像：{ role: "frontend", tech_stack: ["React", "JS"] }
```

#### 路径 3：反馈信号强化
- 用户点赞/踩某条回答 → 调整未来的回答风格
- 用户反复追问 → 推断上一条回答不够详细 → 更新"偏好详细解释"标签

### 5.4 存储结构

```json
{
  "user_id": "u_001",
  "profile": {
    "role": "data_scientist",
    "expertise": { "Python": "expert", "React": "beginner" },
    "language_preference": "zh-CN",
    "response_style": "concise_with_examples"
  },
  "long_term_facts": [
    "正在研究 RAG 系统的日志监控",
    "使用 Pinecone 作为向量数据库"
  ],
  "preferences": {
    "use_emoji": true,
    "prefer_tables": true,
    "avoid": ["过度总结", "重复已知内容"]
  }
}
```

## 六、📊 四类记忆对比总览

| 维度 | 🟡 短时记忆 | 🔵 长时记忆 | 🟢 检索记忆 | 🟣 用户建模 |
|------|------------|------------|------------|------------|
| **触发方式** | 每次对话自动携带 | 蒸馏 / 反思 / 注入 / 反馈 / 标记 | 用户发问时实时检索 | 隐式推断 / 显式收集 |
| **写入时机** | 实时（每条消息） | 对话中/结束时/批量导入 | 离线写入 + 在线检索 | 持续更新 |
| **保留时长** | 当前会话，关窗即失 | 永久（除非主动删除） | 永久（知识库生命周期） | 永久，动态更新 |
| **容量限制** | 受 Token 窗口限制 | 受存储容量限制（理论无限） | 受存储容量限制（理论无限） | 较小（结构化画像） |
| **检索方式** | 无需检索，直接拼入 | 取决于后端：向量检索 / KV 读取 / SQL / 图遍历 | 向量相似度 / 图检索 | 直接读取结构化字段 |
| **典型技术** | Transformer Attention | MemGPT、Reflection | RAG、GraphRAG、HyDE | 用户画像系统、A/B 偏好学习 |
| **适用场景** | 单次任务、短对话 | 知识密集型、企业 QA | 文档问答、知识库查询 | 个性化助手、长期陪伴型 Agent |
| **主要风险** | 上下文污染、token 溢出 | 知识过时、幻觉来源 | 检索不准、上下文截断 | 隐私问题、画像偏差 |

## 七、🔗 完整 Agent 记忆架构

```
用户发问
   │
   ├──→ 🟣 用户建模   → 读取用户画像，调整回答风格和深度
   │
   ├──→ 🟢 检索记忆   → 从知识库 RAG 召回相关上下文
   │
   ├──→ 🔵 长时记忆   → 读取与该用户/主题相关的历史摘要
   │
   └──→ 🟡 短时记忆   → 当前会话上下文
              │
              ↓
        LLM 综合以上四路信息 → 生成回答
              │
              ↓
        会话结束 → 蒸馏精华 → 更新长时记忆 + 用户建模
```

> 💡 **核心思路**：短时记忆是"工作台"，长时记忆是"档案室"，检索记忆是"图书馆"，用户建模是"了解你的私人助理"。四者各司其职，缺一不可。
