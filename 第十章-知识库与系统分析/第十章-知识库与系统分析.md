# 知识库与系统分析

## Overview

> [!NOTE]
> **知识库是 Agent 的外部长期记忆，系统分析是 Agent 理解复杂项目的基础能力。**
> 它们并不是某个具体功能，而是让 Agent 能够真正"记住有用信息、读懂复杂系统"的关键基础设施。
>
> 更直接一点说：
>
> **上下文窗口解决"这次对话里有什么"，知识库解决"跨会话、跨项目的知识怎么组织和调用"。**
>
> 所以，**知识库本质上是在 Agent 系统中，把零散的文档、经验和结构化信息，升级为可检索、可维护、可持续积累的知识资产。**
>
> 而系统分析，则是 Agent 面对一个陌生项目时，能够快速建立认知地图、定位关键模块、理解整体结构的方法论。

## 1. 为什么 Agent 需要知识库

在很多人的直觉里，Agent 的记忆似乎只是"对话历史"。  
但一旦进入真实的长期使用、跨项目协作或知识密集型任务，就会发现：仅仅依赖上下文窗口是不够的，它还必须能够**稳定地存储、检索和调用超出单次对话范围的知识**。

这正是知识库出现的位置。

上下文窗口有三个根本局限：

- **容量有限**：再大的窗口也有上限，无法容纳整个项目的所有文档
- **不可持久**：对话结束后，窗口内容消失，下次对话从零开始
- **无法组织**：原始文本堆叠在窗口里，没有结构，检索效率极低

知识库解决的正是这三个问题：

- **持久存储**：知识独立于对话存在，跨会话可用
- **结构化组织**：知识按主题、类型、层级整理，而不是平铺堆叠
- **按需检索**：Agent 只在需要时取出相关知识，而不是把所有内容都塞进上下文

因此，如果说上下文窗口是 Agent 的工作记忆，那么知识库就是 Agent 的长期记忆。

## 2. WikiLLM：知识库的一种典型形态

### 2.1 什么是 WikiLLM

WikiLLM 不是某个具体产品或行业标准，而是一种**设计模式**——借鉴 Wiki 的组织方式，针对 LLM 的使用特点进行适配。  
它的核心思想是：**以页面为单位、以链接为结构、以协作为机制**，并针对 LLM 的特点做了以下适配：

- 每个知识单元是一个独立的、自包含的文档页面
- 页面之间通过引用和链接形成网状结构
- 内容以 Markdown 为主，便于 LLM 直接读取和生成
- 支持增量更新，而不是每次全量重写

你可以把它简单理解为：

> **WikiLLM 是专门为 LLM 设计的、可持续维护的结构化知识网络。**

### 2.2 WikiLLM 和普通文档的区别

普通文档是给人读的，WikiLLM 是给 LLM 用的。这个区别带来了几个具体的设计差异：

### 2.3 WikiLLM 在 Agent 系统中的位置

WikiLLM 不是 Agent 的执行层，也不是推理层，它是 Agent 的**知识供给层**。

在一个完整的 Agent 系统里，WikiLLM 通常处于这个位置：

```
用户输入
  ↓
Agent 推理（LLM + ReAct）
  ↓ 需要背景知识时
WikiLLM 检索 → 返回相关页面
  ↓
注入上下文 → 继续推理
  ↓
执行动作（CLI / Tool / MCP）
```

它的核心价值是：**让 Agent 在需要的时候，能够取到准确的、结构化的背景知识，而不是依赖模型的参数记忆或把所有文档都塞进上下文。**

## 3. 知识组织：如何让知识可被调用

知识库里有内容，不等于知识可以被有效调用。  
知识组织解决的是：**如何让 Agent 在正确的时机，找到正确的知识。**

### 3.1 分层组织

知识不是平等的，不同层级的知识有不同的调用频率和使用方式。一个合理的分层结构通常包含三层：

**L1 — 索引层（始终可见）**  
类似 WikiLLM 的目录或标签系统。Agent 通过这一层判断"有没有相关知识"，以及"去哪里找"。这一层应该保持精简，控制在几百词以内，始终存在于上下文中。

**L2 — 摘要层（按需加载）**  
每个知识单元的核心内容，通常是一个自包含的 Markdown 页面。Agent 在确认需要某块知识后，加载这一层。内容控制在 2000-5000 词，保证 LLM 能够完整读取。

**L3 — 详情层（深度引用）**  
原始文档、代码、数据、示例等。Agent 通常不直接读取这一层，而是通过脚本或工具按需提取其中的片段。

这个分层逻辑和 Skill 的三层设计（元数据 → 正文 → 资源文件）是一致的：**不是所有信息都需要一开始就加载，让不同的信息在不同的时机进入上下文。**

### 3.2 分类与标签

知识的分类方式决定了检索的效率。常见的分类维度有：

- **主题分类**：按领域或功能模块划分（如"API 文档"、"架构决策"、"错误处理"）
- **类型分类**：按知识形态划分（如"概念解释"、"操作步骤"、"参考示例"、"决策记录"）
- **时效分类**：按更新频率划分（如"稳定知识"、"版本相关"、"临时笔记"）

好的分类不是越细越好，而是**让 Agent 能够用最少的检索步骤找到最相关的内容**。

### 3.3 知识的自包含原则

WikiLLM 中每个页面应该尽量自包含：读完这一页，不需要跳转到其他页面就能理解核心内容。  
这个原则来自 LLM 的工作方式——它不像人类可以随时翻回去看，它只能处理当前上下文里的内容。

因此，写知识库页面时应该：

- 在页面开头给出足够的背景，不假设读者记得其他页面
- 把关键定义和前提条件显式写出来，而不是用"如前所述"
- 跨页引用时，把被引用内容的核心摘要内联进来，而不是只给一个链接

| 人工审核 | 按需 | 关键知识变更需要人工确认 |

## 5. RAG：让 LLM 检索外部知识

RAG是目前最主流的知识库接入方式。  
它的核心思想很简单：**LLM 生成答案之前，先从外部知识库里检索相关内容，把检索结果注入上下文，再让 LLM 基于这些内容生成回答。**

这解决了 LLM 的一个根本问题：参数记忆是静态的，训练截止后就不再更新；而 RAG 让 LLM 能够访问实时的、私有的、超出训练数据范围的知识。

### 5.1 朴素 RAG 的完整流程

一个标准的 RAG 流程分为两个阶段：**索引阶段**和**检索生成阶段**。

**索引阶段（离线，一次性）**

```
原始文档（PDF / Markdown / 网页 / 代码）
  ↓ 文档切分（Chunking）
若干文本块（Chunk）
  ↓ Embedding 模型编码
向量表示
  ↓ 存入向量数据库
向量索引（可检索）
```

**检索生成阶段（在线，每次查询）**

```
用户问题
  ↓ Embedding 模型编码
查询向量
  ↓ 向量相似度检索
Top-K 相关文本块
  ↓ 注入 LLM 上下文
LLM 基于检索结果生成回答
```

### 5.2 RAG 的三个核心组件

**Chunking（文档切分）**  
把长文档切成适合 LLM 处理的小块。切分策略直接影响检索质量：

- 按固定长度切分：简单，但可能切断语义完整的段落
- 按段落/章节切分：保留语义完整性，但块大小不均匀
- 重叠切分（Sliding Window）：相邻块之间有重叠，避免关键信息被切断

**Embedding（向量化）**  
把文本块编码为高维向量，使得语义相近的文本在向量空间中距离更近。常用的 Embedding 模型有 OpenAI text-embedding-3、BGE、E5、Cohere Embed、Voyage AI 等。Embedding 模型的选择直接决定了检索的语义理解能力。

**向量数据库**  
存储和检索向量的专用数据库。常见选择：

| 数据库 | 特点 | 适合场景 |
| --- | --- | --- |
| Chroma | 轻量，本地部署，Python 友好 | 个人项目、快速原型 |
| Qdrant | 高性能，支持过滤，Rust 实现 | 生产环境，需要精确过滤 |
| Weaviate | 内置 Embedding，支持混合检索 | 需要关键词+向量混合检索 |
| FAISS | Meta 开源，纯向量检索，极快，需自行管理持久化 | 大规模离线检索，对延迟敏感的场景 |
| pgvector | PostgreSQL 扩展，SQL 友好 | 已有 PG 数据库的项目 |

### 5.3 朴素 RAG 的局限

朴素 RAG 在很多场景下已经够用，但它有几个明显的局限：

**局限1：单跳检索，无法处理多步推理**  
如果问题需要跨多个文档、多个知识点才能回答，朴素 RAG 一次检索往往只能拿到部分信息。

例如：「A 公司的 CEO 毕业于哪所大学？」  
这需要先找到 A 公司的 CEO 是谁，再找这个人的教育背景——两步检索，朴素 RAG 一次只能做一步。

**局限2：语义相似 ≠ 逻辑相关**  
向量相似度衡量的是语义接近程度，但有时候逻辑上最相关的内容，语义上并不相似。

**局限3：知识孤岛，无法利用实体关系**  
文档被切成独立的块，块与块之间的关系（如"A 是 B 的子类"、"C 依赖 D"）在向量检索中完全丢失。

这三个局限，正是 Graph RAG 和 Agentic RAG 分别要解决的问题。

## 6. Graph RAG：用图结构增强检索

Graph RAG 是对朴素 RAG 的一次重要升级。  
它的核心思想是：**在向量检索的基础上，引入知识图谱来表示实体之间的关系，让检索不只是"找相似文本"，而是"沿着关系图推理"。**

### 6.1 知识图谱是什么

知识图谱（Knowledge Graph）用三元组 `(实体, 关系, 实体)` 来表示知识：

```
(Claude Code, 开发者, Anthropic)
(Anthropic, 发布, Claude 3.5 Sonnet)
(Claude 3.5 Sonnet, 属于, Claude 3.5 系列)
(NanoBot, 使用, Claude API)
```

这种结构的优势在于：**知识不再是孤立的文本块，而是一张可以被遍历和推理的关系网络。**

### 6.2 Graph RAG 的工作方式

Graph RAG 在朴素 RAG 的索引阶段增加了一步：从文档中抽取实体和关系，构建知识图谱。

**索引阶段**

```
原始文档
  ↓ 文档切分 + Embedding（同朴素 RAG）
向量索引
  ↓ 实体抽取 + 关系抽取（LLM 辅助）
知识图谱（实体节点 + 关系边）
```

**检索阶段**

```
用户问题
  ↓ 识别问题中的关键实体
图谱中定位相关节点
  ↓ 沿关系边扩展（1-2 跳）
相关实体及其上下文
  ↓ 结合向量检索结果
综合上下文 → LLM 生成回答
```

### 6.3 Graph RAG 解决了什么

**多跳推理**  
通过在图谱中沿关系边扩展，Graph RAG 可以自然地完成多步推理。「A 公司 CEO 的母校」这类问题，只需要在图谱中走两步：A公司 → CEO → 教育背景。

**全局摘要**  
微软的 [GraphRAG](https://github.com/microsoft/graphrag) 项目提出了一种"社区摘要"机制：把知识图谱中关系密切的节点聚合成社区，为每个社区生成摘要。这让 LLM 能够回答需要全局理解的问题（如"这份文档的主要主题是什么"），而不只是局部检索。

**关系感知**  
当问题涉及实体之间的关系时（如"哪些工具依赖 X"、"X 和 Y 有什么区别"），Graph RAG 能够直接从图谱中提取关系，而不是依赖语义相似度碰运气。

### 6.4 Graph RAG 的代价

Graph RAG 的能力提升是有代价的：

- **构建成本高**：实体抽取和关系抽取通常需要 LLM 辅助，索引阶段的计算成本远高于朴素 RAG
- **维护复杂**：知识图谱需要随文档更新而更新，增量维护比向量索引更复杂
- **不是所有场景都需要**：如果知识库主要是独立文档、问题不涉及跨文档推理，朴素 RAG 已经足够

因此，Graph RAG 更适合：知识之间关系密集、问题需要多步推理、需要全局理解的场景。

## 7. Agentic RAG：让 Agent 主动检索

朴素 RAG 和 Graph RAG 都是**被动检索**：用户提问 → 系统检索 → LLM 回答，整个过程是线性的、一次性的。

Agentic RAG 的升级在于：**把检索变成 Agent 推理循环的一部分，让 Agent 能够主动决定何时检索、检索什么、检索几次，并根据检索结果调整后续行动。**

### 7.2 Agentic RAG 的核心模式

**查询重写**  
用户的原始问题往往不是最好的检索查询。Agentic RAG 中，Agent 会先分析问题，把它改写成更适合检索的形式，或者拆分成多个子查询分别检索。

```
用户问题："Claude Code 和 Codex 在多 Agent 协作上有什么区别？"
  ↓ Agent 拆分
子查询1："Claude Code 多 Agent 协作能力"
子查询2："Codex 多 Agent 协作能力"
  ↓ 分别检索，合并结果
  ↓ LLM 对比生成回答
```

**迭代检索**  
Agent 检索一次后，评估结果是否足够，如果不够则继续检索。这个过程可以循环多次，直到 Agent 认为已经收集到足够的信息。

```
第1轮：检索 → 发现信息不完整
  ↓ 分析缺失的部分
第2轮：针对缺失部分重新检索
  ↓ 信息足够
生成最终回答
```

**自适应检索**  
Agent 根据问题的类型和复杂度，动态选择检索策略：简单问题直接用向量检索，复杂问题切换到 Graph RAG，需要最新信息时调用网络搜索。

**检索-验证循环**  
检索到信息后，Agent 先验证信息的相关性和可靠性，再决定是否使用。如果检索结果与问题不相关，直接丢弃并重新检索，而不是把无关内容塞进上下文。

### 7.3 Agentic RAG 在 Agent 系统中的位置

Agentic RAG 不是一个独立的系统，而是 Agent 推理循环（ReAct）的一个组成部分。  
检索工具和其他工具（CLI、API、文件读写）处于同等地位，都是 Agent 可以主动调用的能力。

```
用户输入
  ↓
Agent 推理（Thought）
  ↓ 判断需要外部知识
调用检索工具（Action）
  ↓
检索结果（Observation）
  ↓
Agent 评估结果，决定下一步（Thought）
  ↓ 信息足够
生成回答 / 执行后续动作
```

这个循环可以重复多次，每次检索都是 Agent 主动发起的，而不是系统自动触发的。

### 7.4 三种 RAG 的选型参考

| 维度 | 朴素 RAG | Graph RAG | Agentic RAG |
| --- | --- | --- | --- |
| 构建成本 | 低 | 高 | 中（依赖 Agent 框架） |
| 检索质量 | 中 | 高（关系密集场景） | 高（复杂问题） |
| 多跳推理 | ❌ | ✅ | ✅ |
| 动态调整 | ❌ | ❌ | ✅ |
| 适合场景 | 独立文档问答 | 实体关系密集的知识库 | 复杂任务、多步推理 |
| 典型工具 | LangChain, LlamaIndex | Microsoft GraphRAG | LangGraph, AutoGen |

三者不是互斥的，实际系统中往往组合使用：用朴素 RAG 处理简单查询，用 Graph RAG 处理关系推理，用 Agentic RAG 协调整个检索过程。

## 8. 上下文管理：如何把知识送进 LLM

有了检索到的知识，下一个问题是：**怎么把它送进 LLM 的上下文，既不超出窗口限制，又能保证相关性？**

### 8.1 注入时机

知识不是越早注入越好，也不是越多越好。上下文管理的核心是**在正确的时机注入正确数量的知识**。

几个常见的注入时机：

- **任务开始时**：注入与任务类型直接相关的背景知识（如项目架构概述、领域术语表）
- **推理过程中**：当 Agent 遇到不确定的概念或需要参考规范时，按需检索并注入
- **工具调用前**：注入与即将调用的工具相关的使用说明和注意事项

注入时应该遵循**最小必要原则**：只注入当前推理步骤真正需要的内容，避免把整个知识库都塞进上下文。

### 8.2 上下文压缩

当知识量超出窗口限制时，需要对上下文进行压缩。常见的压缩策略有：

**摘要替换**：把长文档替换为 LLM 生成的摘要，保留核心信息，丢弃细节。

**滑动窗口**：只保留最近的 N 轮对话和最相关的 K 个知识片段，丢弃较早的历史。

**分级加载**：先加载 L1 索引层，只有在确认需要时才加载 L2 摘要层，极少情况下才加载 L3 详情层。这个分层逻辑和第3节的知识组织直接对应。

### 8.3 Lost in the Middle 问题

研究发现，LLM 对上下文中不同位置的内容注意力分布不均匀：**开头和结尾的内容更容易被注意到，中间的内容容易被忽略**，这个现象被称为"Lost in the Middle"。

实际注入知识时，应该把最重要的内容放在上下文的开头或结尾，而不是随机插入中间。

上下文管理的最终目标是：**让 LLM 在有限的窗口里，始终能看到最有价值的信息，并且这些信息出现在 LLM 最容易注意到的位置。**

## 9. 系统拆解方法：Agent 如何读懂一个项目

知识库解决的是"已知知识如何组织和调用"，而系统拆解解决的是另一个问题：**当 Agent 面对一个陌生的、复杂的项目时，如何快速建立认知地图？**

### 9.1 为什么需要系统拆解

一个真实的软件项目往往包含数十个文件、多个模块、复杂的依赖关系。  
如果 Agent 每次都从头读所有文件，不仅效率极低，还会很快耗尽上下文窗口。

系统拆解的目标是：**用最少的读取成本，建立对项目整体结构的准确认知。**

### 9.2 拆解路径：从入口到模块

一个有效的系统拆解通常遵循以下路径：

**第一步：找入口**  
从项目的顶层文件开始：`README`、`package.json`、`pyproject.toml`、`Makefile`、`main.py` 等。这些文件通常包含项目的整体描述、依赖关系和启动方式。

**第二步：识别边界**  
找到项目的主要模块划分。通常体现在目录结构上：`src/`、`lib/`、`api/`、`core/` 等。每个目录往往对应一个功能边界。

**第三步：定位核心**  
在每个模块内，找到最核心的文件——通常是模块的入口文件、主要类定义或核心算法所在位置。不需要读所有文件，只需要读懂核心文件就能理解模块的主要职责。

**第四步：理解接口**  
模块之间如何通信？通过函数调用、消息队列、HTTP API 还是共享状态？理解接口比理解实现更重要，因为接口决定了模块之间的依赖关系和数据流向。

**第五步：建立地图**  
把以上信息整理成一个简洁的认知地图：项目有哪些模块、每个模块做什么、模块之间如何连接。这个地图不需要完整，但需要准确。

### 9.3 ACP 协议：系统拆解的工程化实践

[ProtoCodeBase](https://github.com/OhMyYuwan/ProtoCodeBase.Skill) 提出了一种基于 ACP（Agent Collaboration Protocol）协议的系统拆解方案。

ACP 协议的核心思想是：**让项目自己描述自己**。

具体来说，ACP 要求每个项目维护两个关键文件：

- **project map**：项目的模块结构、文件职责和依赖关系的结构化描述
- **路由文件**：Agent 进入项目时的导航入口，告诉 Agent 去哪里找什么

有了这两个文件，Agent 不需要从头扫描整个项目，只需要读取 project map 和路由文件，就能快速建立对项目的基本认知，然后按需深入到具体模块。

这个设计的价值在于：**它把系统拆解的成本从"每次进入项目都要重新分析"，降低到"一次维护，多次复用"。**

### 9.4 系统拆解与知识库的关系

系统拆解的结果，本身就是一种知识。  
拆解完成后，把 project map、模块说明、接口文档整理进知识库，下次 Agent 进入同一个项目时，就不需要重新拆解，直接从知识库里取出上次的分析结果即可。

这形成了一个正向循环：

```
系统拆解 → 生成结构化知识 → 存入知识库
                                  ↓
下次进入项目 → 从知识库检索 → 快速建立认知
                                  ↓
                         发现新变化 → 更新知识库
```

**系统拆解是知识库的输入来源之一，知识库是系统拆解成果的持久化载体。**

# 本章小结

知识库和系统分析解决的是 Agent 在"记忆"和"认知"两个维度上的核心问题。

- **WikiLLM** 提供了一种面向 LLM 的知识组织形态，让知识可以被结构化存储和检索
- **知识组织** 通过分层、分类和自包含原则，让知识在正确的时机以正确的粒度进入上下文
- **RAG** 是最主流的知识接入方式，通过向量检索把外部知识注入 LLM 上下文
- **Graph RAG** 在向量检索基础上引入知识图谱，解决多跳推理和关系感知的问题
- **Agentic RAG** 把检索变成 Agent 推理循环的一部分，让 Agent 能够主动、迭代地检索
- **上下文管理** 解决了知识如何高效注入 LLM 的工程问题，核心是最小必要原则和 Lost in the Middle 的规避
- **系统拆解** 让 Agent 面对陌生项目时能够快速建立认知地图，而不是盲目扫描所有文件

从朴素 RAG 到 Graph RAG 再到 Agentic RAG，是一条清晰的能力升级路径：

> **朴素 RAG 解决"能检索"→ Graph RAG 解决"检索得更准"→ Agentic RAG 解决"检索得更聪明"**

这条链路打通之后，Agent 就不再是"每次对话都从零开始"的短期工具，而是能够持续积累、跨项目复用、长期成长的知识型系统。
