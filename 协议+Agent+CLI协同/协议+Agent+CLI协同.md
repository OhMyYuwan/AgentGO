# 协议 + Agent + CLI 协同

## Overview

这一章想讲的是我们的一个核心愿景：

**不要只让 Agent 临时完成一次任务，而是让项目本身变成可以被 Agent 持续理解、持续接手、持续组合的工程对象。**

为了做到这一点，我们把 ProtoCodeBase 分成几层来推进：

```text
协议：定义项目应该如何被理解和组织
Skill：让 Agent 能够遵循协议解析项目；
(.acp)：Agent 根据 Skill 解析项目，并将结果保存在(.acp)文件中，让任何 Agent 都能通过读取(.acp)快速参与项目构建
ProtoCodeBase：解析大量项目并将解析后的文件（.acp）集中存储；
AgenticRAG：能够按照用户功能需求和其他描述，按照解析结果搜索成熟项目中的可参考模块
CLI：将以上特性串联在一起，让可以ProtoCodeBase服务（检索、下载和复用）能够直接被访问；

```

这不是一个单点工具，而是一套协同关系：协议负责统一语言，Skill 负责让 Agent 会用这套语言，AgenticRAG 
负责把协议化项目变成可访问 pipeline，CLI 负责把用户需求和成熟工程实现连接起来。

## 1. 协议：先让项目可以被理解

ProtoCodeBase 的第一层是协议。

我们希望项目不只是一个文件夹，也不只是一次聊天上下文里的临时解释，而是能够按照稳定结构描述自己：

- 这个项目有哪些能力
- 每个能力由哪些模块承载
- 用户需求应该路由到哪里
- 哪些代码可以改，哪些边界需要谨慎
- 一次修改如何从 Request、Plan 到 Change 形成闭环

当项目被 ProtoCodeBase 解析后，它就不再只是“某个仓库里的代码”，而是一个可以被 Agent 读取、路由、复用和继续演化的项目对象。


## 2. Skill：让 Agent 学会遵循协议

协议本身不会自动改变 Agent 的行为，所以我们把协议打包成 Skill：

[ProtoCodeBase.Skill](https://github.com/OhMyYuwan/ProtoCodeBase.Skill)

Skill 的作用是让 Agent 在进入项目时，知道应该如何读取协议、如何理解项目结构、如何生成符合协议的工作计划，以及如何维护项目的协议文件。

也就是说：

```text
协议定义规则
Skill 教会 Agent 按规则工作
```

当一个 CodeAgent 安装了 ProtoCodeBase Skill，它就不再只是“看代码然后猜”，而是可以沿着协议规定的方式进入项目，理解项目，并参与项目修改。


## 3. CLI：把协议化项目变成可访问资源

我们希望 ProtoCodeBase 不只是本地的一套协议文件，也能成为可以被下载、检索和复用的工程资源。

因此，我们会提供 ProtoCodeBase 网站服务和配套 CLI。

我们的设想是：

1. 收集一批成熟项目
2. 使用协议对它们进行解析
3. 保存对应版本的解析结果
4. 将解析结果和原始代码版本关联起来
5. 通过 CLI 下载指定项目、指定版本的解析内容和代码

这样，Agent 不需要每次从头理解一个成熟项目。它可以先拿到协议化后的项目说明，再按需进入源码细节。


## 4. AgenticRAG：用需求检索成熟工程能力

仅仅保存项目解析结果还不够。真正有价值的是：当用户提出一个需求时，系统能帮助 Agent 找到成熟项目里可参考的实现。

这就是 AgenticRAG 的位置。

普通 RAG 更像“从文档里找答案”，而我们希望 AgenticRAG 能面向工程需求工作：

```text
用户需求
  ↓
能力语义检索
  ↓
匹配成熟项目中的模块 / 功能 / 结构
  ↓
返回可参考的代码路径、设计思路和修改方向
  ↓
Agent 结合当前项目生成 Plan 并执行
```

换句话说，我们不是让 Agent 盲目造轮子，而是让它先知道：成熟工业项目里类似能力是怎么组织、怎么实现、怎么扩展的。


## 5. 构建你的私有 ProtoCodeBase

除了使用我们维护的公共解析结果，团队也可以构建自己的私有 ProtoCodeBase。

比如，一个安全研究团队、一个应用开发团队，或者一个企业内部工程团队，都可以把自己的项目进行协议化解析，并用本地 AgenticRAG 建立私有项目知识库。

这样做的意义是：

- 内部代码不需要上传到公共服务
- 团队自己的项目经验可以被沉淀
- Agent 可以在多个内部项目之间迁移经验
- 新人和新 Agent 都能更快接手已有项目

## 6. 从驾驭 Agent 到驾驭 Project

最后，我们真正想推动的是一个视角变化：

```text
从驾驭 Agent
到驾驭 Project
```

使用 Agent 的最终目标，不只是让它回答问题，也不只是让它完成一次代码修改，而是让它真正参与项目的持续构建。

而驾驭项目，意味着：

- 项目能力可以被清晰描述
- 项目结构可以被快速理解
- 项目修改可以被计划和追踪
- 不同 Agent 可以接力同一个项目
- 成熟项目的能力可以被迁移和组合

我们希望未来的项目像自来水一样：能力不是封闭在某个仓库、某个人、某次聊天里，而是可以顺着协议流动，灌进每个用户的需求水瓶。

这就是 ProtoCodeBase 想做的事情。
