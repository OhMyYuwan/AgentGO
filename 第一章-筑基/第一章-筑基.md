本章是整个 Agent 学习库的起点。与其一开始就进入复杂的多 Agent 协作、记忆系统或项目级协议，不如先回答几个更基础也更关键的问题：今天常见的 Agent 开发入口有哪些？IDE、CLI 与桌面 Agent 都有什么？一个真正可持续使用的 Agent 系统由哪些部分构成？Skill 为什么会成为现代 Agent 生态中的重要组织单元？

## Agentic Coding 的入口
认识当前主流的 agentic coding 入口，包括 [Claude Code](https://code.claude.com/docs/zh-CN/vs-code)、[Codex](https://openai.com/codex/) 与 [OpenCode](https://opencode.ai/)。它们共同代表了一类新的开发方式：模型不再只是回答代码问题，而是能够直接读代码库、修改文件、执行命令，并在 IDE、终端或桌面环境中参与真实的软件开发流程。Claude Code 官方文档已经把它定义为可以读代码库、编辑文件、运行命令并与开发工具集成的 agentic coding tool；OpenCode 也明确强调自己是 open source AI coding agent，支持 terminal、desktop 与 IDE extension；Codex 则进一步把多 Agent、并行工作流与长任务协作带入了桌面界面。

## Agentic Coding 工具
[CC-Switch](https://github.com/farion1231/cc-switch) 解决了一个很现实的问题：当你同时使用多个 coding agent 或不同 provider 时，如何统一切换和管理配置。对初学者来说，这一层虽然不起眼，却往往决定了整个环境是否顺手、是否稳定。CC-Switch 官方当前支持 Claude Code、Codex、Gemini CLI、OpenCode 和 OpenClaw，并把 provider presets 和 configuration management 作为主要能力之一。CC-Switch 可以帮你检查是否完整安装了本项目需要的Agentic Coding的 CLI 以及 APP。

##  Agent 系统
我们认为有三个具有代表性的 Agent 系统：[NanoBot](https://github.com/HKUDS/nanobot)、[OpenClaw](https://github.com/openclaw/openclaw) 和 [Hermes Agent](https://github.com/nousresearch/hermes-agent)。之所以选择这三个项目，是因为它们分别对应了三种不同的理解路径。NanoBot 强调轻量和最小化，适合用来理解 Agent 的基础骨架；OpenClaw 强调“你自己运行在设备上的个人 AI 助手”，适合理解消息通道、控制平面、常驻 Agent 和真实生活接入；Hermes Agent 则强调学习循环、记忆、skills 与跨会话持续成长，更接近“会积累经验的 Agent”。因此，本章并不会急于比较谁更强，而是先帮助读者形成对 Agent 类型和设计取向的基本认识。

## Skill
Skill 是 LLM --> Agent 的一个关键过程。如果没有 Skill，很多 Agent 系统最后都会退化成“巨型 prompt + 一堆工具”的混合体；而有了 Skill 以后，能力就可以被整理成可安装、可发现、可复用的模块。[skills.sh](https://skills.sh/) 直接把 Skills 描述为“reusable capabilities for AI agents”，并支持单命令安装；Anthropics 提供的 skill-creator 则进一步说明，Skill 不只是静态文档，还可以被系统化地创建、测试和迭代。对学习者而言，这意味着后面学到的方法、规则和工作流，不必每次都重新塞进 prompt，而可以逐渐沉淀为 Agent 的长期能力资产。Skill 的来源分为三类：1. [公开来源](https://skills.sh/) 2. [自行总结](https://skills.sh/anthropics/skills/skill-creator) 以及 海量数据生成技能([同事](https://github.com/titanwings/colleague-skill)，[女娲](https://github.com/alchaincyf/nuwa-skill))。

## MCP/CLI
MCP 和 CLI 是 Agent 执行的手，他们的执行的结果就是 Agent 对世界的行动的观察。我们认为比较有效的三个库首先是 CLI 构建库[OpenCLI](https://github.com/jackwener/opencli) 用来创建任何开源代码的 CLI；然后是 MCP 的官方库 [Model Context Protocol Docs](https://modelcontextprotocol.io/docs/getting-started/intro) + MCP 服务器搭建 [Servers](https://github.com/modelcontextprotocol/servers)。要说推荐还是推荐 CLI + Skill 我们现阶段接触不到必须要 MCP 实现的内容。

## ProtoCodeBase
ProtoCodeBase 提供了一种构想，就是软件像水一样流动。现在大量教程实际上是在让 Agent 造轮子，应该提出一种协同一致的方法让 Agent 能够在不同项目之间进行快速解析和借鉴、让大量 Agent 能够在项目中进行无人工干预的流水线协同。这部分需要安装 [ProtoCodeBase.Skill](https://github.com/OhMyYuwan/ProtoCodeBase.Skill)，了解一下项目方案即可，目前我们需要知道的只有使用 ACP 协议解析这个项目，并返回我们项目解析报告，了解这个 ACP 项目的情况以及遵循 ACP 协议进行开发，保证 project map 以及 路由文件得到稳定维护。然后就可以丝滑地使用不同模型在同一个项目上快速接手了。
