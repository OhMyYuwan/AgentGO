# 第三章 MCP/CLI

## Overview

> [!NOTE]
> **CLI 是 Agent 与外部环境交互的基础执行接口，MCP 是面向 Agent 的标准化能力接入协议。**
> 它们并不是某个具体功能，而是让 Agent 能够真正“连上外部世界、调用实际能力”的关键基础设施。
>
> 更直接一点说：
>
> **CLI 是 Agent 的独立执行接口**，Skill 是 CLI 的方法封装方法。
> 二者结合后，已经能够支撑大量真实的 Agent 使用场景：Skill 负责定义“这类事情通常该怎么做”，CLI 负责把脚本、命令和本地能力真正执行出来。
> 
> **MCP 是另一种能力接入方式**。它适合需要**统一**描述、跨客户端共享、远程服务接入的场景，负责把这些外部能力整理成 Agent 更容易理解和调用的统一接口；但它并不是 CLI 的前置条件，也不是 CLI 的升级终点。
> 
> 目前的实践来看，MCP 正在系统系地衰落，仅在一些工业化场景中进行使用。在我们的日常使用中 **MCP 正被 CLI+Skill** 吊打。在大量本地自动化、编码辅助和个人工作流场景中，`CLI + Skill` 本身就是一条完整、稳定、低成本且高可控的 Agent 路线。

## 1. 为什么 Agent 需要 CLI 与 MCP

在很多人的直觉里，Agent 的核心似乎只是“大模型 + Prompt + Tool”。  
但一旦进入真实的软件开发、自动化执行或项目协同场景，就会发现：仅仅让模型“会回答”是不够的，它还必须能够**稳定地调用命令、连接外部系统、读取服务能力，并在复杂环境里完成真实操作**。

这正是 CLI 与 MCP 出现的位置。

- **CLI** 解决的是“如何真正执行”
- **MCP** 解决的是“如何把外部能力以统一方式交给 Agent 使用”

因此，如果说 Prompt 和 Skill 主要解决“怎么思考、怎么做”，那么 CLI 和 MCP 解决的就是“如何真正落地执行”。

---

## 2. CLI 是什么

CLI（Command-Line Interface，命令行界面）本质上是一种**通过文本命令操作程序和系统的接口**。  
对于普通开发者来说，CLI 是终端里输入命令、运行脚本、查看结果的方式；而对于 Agent 来说，CLI 则是它接触本地环境最基础的执行入口之一。

在 Agent 系统里，CLI 通常承担以下几类职责：

- 调用本地脚本
- 读写文件与目录
- 运行构建、测试、部署命令
- 管理配置
- 安装、列出、移除插件、Skill 或 MCP server
- 启动和控制其他程序或服务

所以，CLI 的重点不是“理解能力”，而是“执行能力”。

你也可以把它简单理解为：

> **CLI 是 Agent 动手做事时最基础的操作面。**

---

## 3. MCP 是什么

MCP（Model Context Protocol）是一种面向 Agent / LLM 的**标准化能力接入协议**。  
它不是某一个工具，也不是某一个插件市场，而是一套统一规则，用来描述：

- 有哪些外部能力可用
- 每个能力叫什么
- 它接收什么参数
- 返回什么结果
- Agent 应该如何发现和调用它们

从协议层看，MCP 当前主要围绕 **tools、resources、prompts** 三类原语展开，并通过标准化的客户端—服务端交互来完成能力发现与调用。

所以，MCP 更像一种“让 Agent 看见并调用外部能力的统一语言”。

你可以把它概括成一句话：

> **CLI 是执行接口，MCP 是接入协议。**

如果你准备进一步学习 MCP，可以从官方 SDK 和成熟 server 入手，而不是一上来自己设计协议。

| 类型 | 项目 | 适合学习什么 |
| --- | --- | --- |
| 官方文档 | [Model Context Protocol Docs](https://modelcontextprotocol.io/docs/getting-started/intro) | MCP 的基本概念、架构、server/client 关系 |
| TypeScript SDK | [modelcontextprotocol/typescript-sdk](https://github.com/modelcontextprotocol/typescript-sdk) | 用 TypeScript 编写 MCP Server |
| Python SDK | [modelcontextprotocol/python-sdk](https://github.com/modelcontextprotocol/python-sdk) | 用 Python 编写 MCP Server |
| GitHub MCP | [github/github-mcp-server](https://github.com/github/github-mcp-server) | 学习成熟服务如何封装 GitHub 能力 |
| 示例 Server | [modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers) | 参考文件系统、Git、数据库等常见 server 形态 |

初学时可以先观察这些 server 的共同结构：

- 它暴露了哪些 tools
- 每个 tool 的参数是什么
- tool 返回的数据是否结构化
- 是否需要鉴权或环境变量
- 是否限制了文件路径、仓库范围或访问权限
- Agent 调用失败时是否有清晰错误信息

这些问题比“代码怎么写”更重要，因为 MCP 的关键不只是跑起来，而是让 Agent 安全、稳定、可控地调用。


---

## 4. CLI 与 MCP 的关系

CLI 和 MCP 经常一起出现，但它们并不在同一层。

### CLI 更像“执行面”
它关注的是：

- 某个命令怎么跑
- 某个脚本怎么执行
- 某个程序怎么启动
- 某个配置怎么管理

### MCP 更像“接入面”
它关注的是：

- 这个外部能力怎么描述
- Agent 怎么发现它
- 它的参数是什么
- 它应该如何被标准化调用

所以它们的关系不是谁替代谁，而是：

- **CLI 偏向“把动作做出来”**
- **MCP 偏向“把动作整理成 Agent 能调用的标准接口”**

但是 CLI 和 MCP 不是绑定关系，也不是先后升级关系。

### CLI 是独立成立的
CLI 本身就是一种完整的执行接口。  
只要 Agent 能够稳定调用命令、脚本和本地程序，CLI 就已经可以直接承担真实任务执行。

尤其当 CLI 与 Skill 结合时：

- Skill 负责定义什么时候用、怎么用、注意什么
- CLI 负责执行脚本、命令、检查和返回结果

这时，Agent 已经具备了“方法封装 + 确定执行”的完整能力闭环。

### MCP 是按需引入的
MCP 并不是 CLI 的必经阶段，也不是所有能力都值得走向 MCP。  
它更适合这些场景：

- 一个能力需要被多个 Agent / IDE / 客户端共享
- 一个能力需要标准化描述参数与返回结构
- 一个能力本身是远程服务，而不是本地脚本
- 一个团队希望统一管理外部能力接入方式

所以，更准确的关系应该是：

> **CLI 是独立执行面，Skill 是方法封装层，MCP 是按需引入的协议化接入层。**

很多实际能力一开始只是一个本地脚本或命令；  
而当你希望这个能力能够被不同 Agent、不同客户端、不同 IDE 更稳定复用时，它就可能进一步被包装成 MCP server 暴露出来。在很多实际场景里，**CLI + Skill 已经足够好用，甚至往往比 MCP 更轻、更稳、更容易调试。**

---

## 5. MCP、Tool、Skill、Workflow 的关系

这几个概念分属不同层级，它们并不是替代关系。

### Tool
Tool 是一个具体动作。  
它回答的是：

> 我现在能做什么操作？

例如：
- 搜索网页
- 读取文件
- 调用 API
- 执行命令

---

### Skill
Skill 是某类任务的做事方法封装。  
它回答的是：

> 这类事情通常该怎么做？

它可以包含：
- 使用规则
- 触发条件
- 操作步骤
- 示例
- 可调用脚本
- 资源文件

Skill 可以直接调用 CLI，也可以调用其他 Tool。

---

### Workflow
Workflow 是全局任务的流程编排。  
它回答的是：

> 整件事怎么一步一步完成？

它关注：
- 步骤顺序
- 分支条件
- 状态流转
- 失败重试
- 模块协作

---

### MCP
MCP 不是 Skill，也不是 Workflow，更不是 CLI 的必选搭档。  
它只是把外部能力以标准化方式暴露给 Agent 的协议层。

它回答的是：

> 某个外部能力如何被统一描述、发现和调用？

因此：

- Tool 是动作
- Skill 是方法
- Workflow 是流程
- MCP 是一种可选的协议化接入方式

在很多本地场景中，**Skill 直接调用 CLI 就已经可以很好地完成任务，并不需要再额外经过 MCP。**

---

## 6. 主流 Agent 中的 CLI / MCP 形态

从 2026 年的主流 Agent / Coding Agent 生态来看，CLI 和 MCP 已经成为非常常见的基础设施。

### Claude Code
Claude Code 将 MCP 作为连接外部工具的重要标准方式，并提供了命令行与会话内入口来管理 MCP server。官方文档还明确提到，MCP 工具规模过大时会带来上下文压力，因此引入了更细粒度的工具发现与控制机制。

### Codex
Codex 将 MCP 放在 customization 体系的重要位置。它既支持在 CLI 中配置 MCP，也支持在 IDE 扩展中复用配置，说明 MCP 已经不只是实验性功能，而是 Agent 外部能力接入的正式入口之一。

### GitHub Copilot CLI / coding agent
Copilot CLI 已经内置 GitHub MCP server，并支持通过 `/mcp add` 等方式添加更多 server；同时，GitHub 官方也明确说明过多 MCP 工具会影响上下文和选择质量，因此建议启用更少、更精确的 toolsets。

### OpenCode
OpenCode 在 CLI 中直接提供 `mcp add/list` 一类命令，并明确提醒：MCP server 会增加上下文占用，工具过多可能直接挤爆上下文窗口。这个提醒非常具有代表性，因为它说明 MCP 的问题不在“能不能接”，而在“接进来之后是否仍然高效”。

---

## 7. 什么时候用 CLI，什么时候用 MCP

### 更适合直接用 CLI 的场景
- 只是本地单机使用
- 目标能力已经是稳定命令或脚本
- 使用者主要是人，而不是多个 Agent 客户端
- 更看重简单直接、低开销、易调试
- 工具不需要标准化共享给多个系统

### 更适合用 MCP 的场景
- 希望将能力标准化提供给多个 Agent / IDE / 客户端
- 需要统一描述 Tool 的名称、参数与返回结构
- 需要远程接入服务、资源或工具集
- 需要在团队或项目中共享同一套能力接入方式
- 需要更规范的发现、认证、权限与接入逻辑

所以你可以这样判断：

> **如果只是把能力跑起来，CLI 往往更直接；如果希望能力被稳定接入和复用，MCP 更有价值。**

---

## 8. 本章小结

CLI 和 MCP 不是一个层面的东西。

- **CLI** 是 Agent 的基础执行接口，负责把命令、脚本和本地环境连接起来
- **MCP** 是 Agent 的标准化能力接入协议，负责把外部工具、资源与服务整理成统一可调用的接口

进一步看：

- **Tool** 是具体动作
- **Skill** 是做事方法
- **Workflow** 是全局流程
- **MCP** 则是让 Tool 能够被标准化交付给 Agent 的接入层

因此，CLI 与 MCP 并不是“谁更先进”的关系，而是 Agent 从“会理解任务”走向“能稳定执行任务”时不可绕开的两层基础设施。

## 附录：CLI *vs.* MCP —— 为什么越来越多场景重新偏向 CLI + Skill

### 1. 先说结论

MCP 不是不能用，而是并不是所有场景都值得 MCP 化。  
对于大量本地自动化、编码辅助和个人工作流来说，**CLI + Skill** 往往已经是更直接、更稳定、更低成本的方案。

因此，问题不在于“MCP 有没有价值”，而在于：

> **很多能力根本不需要经过 MCP，直接由 Skill 调用 CLI 就足够了。**

---

### 2. CLI + Skill 的优势

#### 2.1 更直接
CLI 本来就是命令、脚本和程序的原生执行方式。  
Agent 不需要额外绕一层协议，就可以直接完成操作。

#### 2.2 更容易和 Skill 配合
Skill 负责：
- 什么时候调用
- 按什么步骤调用
- 调用前后检查什么
- 如何组织输入输出

CLI 负责：
- 真正执行脚本和命令

这两者天然形成闭环。

#### 2.3 更容易调试
CLI 的问题通常更容易定位：
- 命令有没有跑
- 参数对不对
- 输出是什么
- 返回码是多少

而 MCP 往往还会多出：
- server 生命周期
- transport 配置
- schema 描述
- 工具发现
- 客户端兼容

#### 2.4 更低上下文负担
CLI 本身不要求额外描述一整套协议对象。  
而 MCP 往往需要维护工具描述、参数 schema、能力暴露信息，这些都会增加上下文与系统复杂度。

#### 2.5 更适合本地能力
如果能力本来就是：
- 本地命令
- 本地脚本
- 本地构建工具
- 本地测试链路
- 本地目录与文件操作

那么直接走 CLI 往往最自然。

---

### 3. MCP 的局限，不在于“不能用”，而在于“并不总划算”

MCP 的主要问题不是无效，而是：

- 它会引入额外描述与管理成本
- 工具一多，模型更难选准
- 部署、认证、调试复杂度更高
- 协议化并不自动等于更好用
- 对本地、单机、单人工作流来说，经常是过度设计

所以，对于很多能力来说：

> **CLI 是原生执行方式，Skill 是方法封装方式，二者结合已经形成完整方案；MCP 只是按需附加，而不是默认必选。**

---

### 4. 最终判断

因此，本教程采取的实践立场是：

> **默认优先考虑 CLI + Skill。**
>  
> 当一个能力本来就是命令、脚本或本地工具时，优先让 Skill 直接组织并调用 CLI。  
> 只有在确实需要跨客户端共享、远程服务接入、统一协议化管理时，再考虑引入 MCP。

一句话总结：

> **不是 CLI 需要 MCP，而是有些能力在特定场景下才值得被 MCP 化。**
